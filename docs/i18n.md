# i18n — internationalization

> Stack: **next-intl 4** + **Next 16 App Router** + **type-safe `t()` via module augmentation**.
> Locales: `vi` (default, no URL prefix) + `en` (prefix `/en/...`).
> Last reviewed: 2026-06-03.

## Layout

```
src/i18n/
├─ routing.ts        # locales, defaultLocale, localized pathnames
├─ navigation.ts     # type-safe Link / useRouter / usePathname / redirect
├─ request.ts        # per-request config (messages + formats + timezone + now)
├─ formats.ts        # Intl presets (currency VND/USD, dates, numbers, list)
├─ config.ts         # TIMEZONE, LOCALE_LABELS, CURRENCY_BY_LOCALE
├─ types.d.ts        # AppConfig module augmentation (Locale + Messages + Formats)
└─ messages/
   ├─ index.ts       # typed `loadMessages(locale)` merger
   ├─ vi/{Common,Home,Errors}.json   # SOURCE OF TRUTH for types
   └─ en/{Common,Home,Errors}.json   # must mirror vi shape
```

The `[locale]` route segment lives at `src/app/[locale]/`. Every layout/page in
that segment must call `setRequestLocale(locale)` so SSG and `getTranslations`
work — even for layouts that only render providers. `hasLocale()` narrows the
type from `string` to `Locale` and 404s on unknown values.

## Conventions

1. **`vi/` is the source of truth.** TypeScript derives the `Messages` type from
   `vi/*.json` (see `types.d.ts`). Missing keys in `en/` cause TS errors.
2. **Namespace per file.** Each top-level UI domain gets its own JSON
   (`Home.json`, `Common.json`, `Errors.json`). Adding a namespace:
   - Create `vi/<Name>.json` and `en/<Name>.json` with the same shape.
   - Add `<Name>` to the `Messages` type in `types.d.ts`.
   - Add `<Name>` to the `NAMESPACES` array in `messages/index.ts`.
3. **One source of formatting.** Never call `new Intl.NumberFormat(...)` in a
   component — use `format.number(price, 'currency')` from
   `useFormatter()` / `getFormatter()`. New presets go in `formats.ts`.
4. **Locale-aware links only.** ESLint forbids `next/link` and
   `next/navigation`'s `redirect|useRouter|usePathname` in `src/`. Import from
   `@/i18n/navigation` instead.
5. **Pathnames** (localized URLs like `/products` ↔ `/san-pham`) live in
   `routing.ts`. Dynamic segments require object-form `<Link href={...}>` and
   are added when the route lands.
6. **Server vs client translations**:
   - Server (RSC, metadata, route handlers): `await getTranslations(...)`
   - Client: `useTranslations(...)`
   - Both share the same key namespace and the same type augmentation.

## Common tasks

### Add a new string

1. Add the key to `src/i18n/messages/vi/<Namespace>.json`.
2. Add the matching key to `en/<Namespace>.json` (TS will complain otherwise).
3. Use `t('Namespace.key')` — autocomplete works.

### Add a new namespace

1. Drop `vi/<Name>.json` + `en/<Name>.json`.
2. Extend `types.d.ts`:
   ```ts
   import type NewNs from './messages/vi/<Name>.json';
   // …
   type Messages = { Common: …; <Name>: typeof NewNs; … };
   ```
3. Extend `messages/index.ts` `NAMESPACES` array.

### Add a new locale

1. Add to `routing.ts` `locales`.
2. Create `messages/<locale>/` mirroring vi/ shape.
3. Update `config.ts` (`LOCALE_LABELS`, `CURRENCY_BY_LOCALE`).
4. Add localized pathname variants in `routing.ts` `pathnames`.

### Rich text with components (links, bold)

Use `t.rich('key', { tag: (chunks) => <a>{chunks}</a> })` with `<tag>` syntax in
the JSON message, not `{tag}` placeholder syntax.

## Type augmentation — how `t()` is typed

`src/i18n/types.d.ts` augments next-intl's `AppConfig`:

```ts
declare module 'next-intl' {
  interface AppConfig {
    Locale: 'vi' | 'en';
    Messages: {
      Common: typeof Common;
      Home: typeof Home;
      Errors: typeof Errors;
    };
    Formats: typeof formats;
  }
}
```

Result: `t('Home.title')` autocompletes; `t('Home.titl')` fails TS. Locale
params are narrowed to `'vi' | 'en'` everywhere.

If you add a namespace and forget to update `types.d.ts`, runtime works but you
lose autocomplete + drift detection — **always update both**.

## When to move `messages/` outside `src/`

Default: **keep in `src/i18n/messages/`**. The current setup co-locates config
with translations and uses clean relative imports.

Move to top-level `/messages/` when **any** of these is true:

| Trigger                                 | Why                                                             |
| --------------------------------------- | --------------------------------------------------------------- |
| ≥ 3 locales                             | Top-level navigation becomes friction-saving                    |
| Integrating Crowdin / Tolgee / Lokalise | Tools default-scan repo root                                    |
| Onboarding non-dev translators          | They expect content, not `src/`                                 |
| Messages > ~1000 keys / locale          | Separate "content" from "code" so CI can skip i18n-only changes |

### Migration recipe (~15 min, one PR)

```bash
git mv src/i18n/messages messages
```

Then edit three files:

- **`src/i18n/types.d.ts`** — adjust relative paths:
  ```ts
  import type Common from '../../messages/vi/Common.json';
  ```
- **`src/i18n/messages/index.ts`** — move to `src/i18n/load-messages.ts` and
  update dynamic import:
  ```ts
  (await import(`../../messages/${locale}/${ns}.json`)).default;
  ```
- **`tsconfig.json`** — add `"messages/**/*.json"` to `include` if needed.

If integrating a translation service, document its config alongside:

```yaml
# crowdin.yml
files:
  - source: /messages/en/*.json
    translation: /messages/%two_letters_code%/%original_file_name%
```
